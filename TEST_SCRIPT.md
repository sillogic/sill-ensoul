# sova 全特性测试脚本

> **用法**：新建两个空项目目录（A、B），各开一个 zcode 会话。按下面顺序把每段**提示词**
> 复制到对应会话，等它执行完，对照"预期结果"打勾。全部测试不改任何代码、不动配置。
>
> **前置**：确保 `sova-mcp` 已 `pip install -e .` 且 zcode 用户级配置 `~/.zcode/cli/config.json`
> 里 `{"command":"sova-mcp"}`。重启过 zcode 让新代码生效。

---

## 项目 A：唤醒 · 自我认知 · 经验召回 · 沉淀

> 在项目 A 的会话里依次贴。每贴一段等执行完再贴下一段。

### 测试 A1 — 唤醒 + 自我认知（你是谁）

```
唤醒 algo-engineer 这个 Agent。然后告诉我：你是谁？你擅长什么？
```

**预期**：
- [ ] 调用了 `agent_index(agent_id="algo-engineer")`
- [ ] 回答引用了 persona：AI 算法工程师，强项含深度学习/推荐系统/LLM 微调
- [ ] **没有**凭空编造 persona 里没有的强项（如"我擅长前端"）

---

### 测试 A2 — 召回历史经验（带经验进项目）

```
我想启动一个新的推荐系统项目，用双塔模型做召回。你之前在这块踩过什么坑？
```

**预期**：
- [ ] 调用了 `wiki_search`（query 含"推荐系统"/"双塔"之类）
- [ ] 调用了 `wiki_read` 读 `projects/demo-project`
- [ ] **引用了真实经验**：负采样比例 1:4 时 AUC 最高，1:10 反而过拟合
- [ ] 不是凭空编的——能说出 concept 来源

---

### 测试 A3 — 查询项目清单 + 某项目细节

```
你做过哪些项目？挑一个详细说说你总结的经验。
```

**预期**：
- [ ] 列出 concept 清单里 `type: Project` 的条目（至少 demo-project）
- [ ] 选一个 `wiki_read` 读出来，引用"关键决策/教训/可复用资产"
- [ ] **清单里不含 `AGENT`**（persona 不污染，H7）

---

### 测试 A4 — 沉淀新经验（提醒式半自动）

```
我们这次用双塔做召回，但发现了一个新问题：当用户行为序列很长时，
双塔的 user tower 输入过长导致推理延迟到 200ms，线上扛不住。
我们的解法是把长序列先做一次 attention pooling 压成固定长度，
延迟降到 50ms，AUC 只掉 0.3%。
```

贴完上面这段后，**不要主动说"记下来"**，看 agent 会不会自己提醒。

**预期**：
- [ ] agent **主动**识别这是可复用经验（非平凡坑 + 解法）
- [ ] 提出沉淀，先问用户"要我记进 wiki 吗"（或类似确认请求）
- [ ] **没有**未经确认就直接写入

然后你回复"记下来吧"：

**预期**：
- [ ] 调用 `wiki_write_concept`（type 必填，应是 `Project` 或 `Reference`）
- [ ] body 是提炼（教训+解法+权衡），**不是**把你那段话原文照搬
- [ ] 配套调用 `wiki_append_log` 记一笔

---

### 测试 A5 — skill 调度

```
这个项目做完想出一份模型评估报告，PDF 格式。
```

**预期**：
- [ ] 检索到 skill 经验（`expertise/skill-pdf`，或提到推荐用 PDF skill）
- [ ] **判断当前环境有没有**该 skill：
  - 如果有 → 提议用它
  - 如果没有 → 提醒你自己装（**不代装、不探测**）
- [ ] 引用了记忆里的用法要点（如"报告类首选""公式密集走 LaTeX 子模式"）

## 项目 B：跨项目记忆留存

> 切到项目 B 的会话。这里测的是核心卖点：**A 项目学的东西，B 项目能用到**。

### 测试 B1 — 唤醒后检索 A 沉淀的新经验

```
唤醒 algo-engineer。我在做一个推荐系统，双塔召回，但线上推理延迟太高，
user tower 输入太长。你之前遇到过这类问题吗？有没有经验？
```

**预期**：
- [ ] 调用 `agent_index` 唤醒
- [ ] 调用 `wiki_search`（query 含"双塔"/"延迟"/"user tower"等）
- [ ] **命中 A4 沉淀的新经验**（attention pooling 压长序列，延迟 200ms→50ms，AUC 掉 0.3%）
- [ ] 这是关键断言：B 项目能读到 A 项目刚写入的经验 → 记忆跨项目留存

---

### 测试 B2 — LoRA 经验（验证较早的沉淀也在）

```
另外我还想用 LoRA 微调一个 LLM 做推荐文案生成。LoRA 的 rank 怎么设？
target_modules 要挂哪些？数据集只有几百条，用 QLoRA 行吗？
```

**预期**：
- [ ] `wiki_search` 命中 `expertise/lora-finetuning`
- [ ] 回答含：r=16/alpha=32 起步；7 个 proj 全挂（含 gate/up/down）；小数据集别用 QLoRA 用 bf16
- [ ] 引用的是记忆里的真实条目，不是凭空编

---

### 测试 B3 — 跨 Agent 隔离

```
唤醒 backend-engineer。它知道 LoRA 微调的经验吗？
```

**预期**：
- [ ] 调用 `agent_index` 切换到 backend-engineer
- [ ] 检索后**找不到** LoRA 经验（那是 algo-engineer 的，隔离）
- [ ] **明确说"我的记忆里没有相关记录"**，不假装有、不替 algo-engineer 回答
